# Чек-лист ревизии логики (mixture-DoE / MoE-GP / ветки)

## Блок 0. Главный инвариант архитектуры

> Это сердце всего. Если здесь «нет» — остальное вторично.

- [ ] **Модель физики ОДНА на проект**, не по одной на ветку. Поиск по коду: нет ли `branch.model`, `branch.surrogate` как owned-объекта?
- [ ] Суррогаты обучаются на **объединённой базе** `X = ⋃ X_b`, а не на точках одной ветки. Проверить вход `fit()`: туда подаётся общая база?
- [ ] Точка имеет **тег происхождения** (какая ветка добрала), но физически лежит в **одной общей базе** — не две копии. Проверить: нет дублирования точек при добавлении.
- [ ] Прогнозы внутри ветки берутся из **общих** суррогатов (`project.surrogates[prop]`), а не из локальной копии.
- [ ] Ветка хранит: цель, ограничения, бюджет, **историю своего добора (теги/ссылки)**, статус, рецепт. Ветка **НЕ** хранит модель.

## Блок 1. Геометрия и численная гигиена (M1)

- [ ] Работа идёт в **псевдокомпонентах / $q-1$ координатах**, не в сырых $q$ с $\sum x_i=1$. Проверить, что в `XᵀX` и GP-ядро не уходят избыточные координаты.
- [ ] Ограничения $L_i \le x_i \le U_i$ реально применяются при генерации вершин (не только в UI).
- [ ] Все кандидатные точки и оптимумы **лежат на симплексе** (проверка $\sum x_i = 1 \pm \epsilon$, $x_i \ge 0$) — есть assert/валидатор?
- [ ] Есть jitter $+\epsilon I$ при обращении $K$ / решении нормальных уравнений.

## Блок 2. Дизайн экспериментов (M2, M5)

- [ ] **Модель — вход генератора дизайна**, а не зашита. Можно ли явно передать `linear / special-quadratic / quadratic`? (наша первая грабля — алгоритм упирался в квадрат).
- [ ] **D-optimal для screening (M2)**, **I-optimal для уточнения (M5)** — разные критерии, не один на всё.
- [ ] Coordinate-exchange — **batch**, не greedy sequential по одной точке. Проверить, что весь дизайн оптимизируется совместно.
- [ ] **Multiple restarts** в оптимизаторе дизайна (многомодальность). Сколько? Фиксирован ли seed для воспроизводимости?
- [ ] Контроль $n$ vs $p$: предупреждение при $n \approx p$ (иначе $R^2=1$ обманывает). Формула $p_{quad}=q+\binom{q}{2}$ считается и показывается?

## Блок 3. Screening и снижение размерности (M3)

- [ ] Scheffé-fit **на каждое свойство** отдельно (не один общий).
- [ ] ARD-lengthscales используются как индикатор значимости компонента (большой $\ell_i$ → неважен).
- [ ] Сводка «компонент → на какие свойства влияет» строится по всем P свойствам.
- [ ] $q_{eff}$ — результат screening, не хардкод. Дальнейшие стадии работают на сокращённом наборе?

## Блок 4. Кластеры / режимы (M4)

- [ ] GMM кластеризует в **пространстве свойств (output)**, не рецептур (input). Это критичное разделение из самого начала.
- [ ] Прообраз режима в пространстве рецептур получается **через модель**, не по евклидовой близости точек (прообраз может быть несвязным).
- [ ] Выбор $K$ по **BIC**, не вручную.
- [ ] **Multiple restarts EM** (многомодальность правдоподобия).
- [ ] Якорные точки **вне кластеров** собираются (иначе модель не знает границ режимов).

Держи дополнение к чек-листу — оформил как новый блок в той же логике (проверка, что не протекает), плюс короткий диагностический протокол под твой конкретный симптом «нужно ещё столько же точек».


## Блок 5.5. Объём и остановка базового I-дизайна

### 5.5.1 Размерность — главная причина «много точек»

- [ ] **M5 работает на $q_{eff}$ (после screening M3), а НЕ на полном $q$.** Проверить вход генератора: сколько компонентов реально в модели?
- [ ] Число параметров считается и логируется: $p_{quad} = q_{eff} + \binom{q_{eff}}{2}$. Сверить с тем, что показывает система.
- [ ] Если M5 крутится на полном $q$ (напр. 12) → это **ошибка размерности**, а не I-критерия: $p_{quad}(12)=78$ честно требует много точек. Фикс — в связке M3→M5, не в M5.

> Быстрый ориентир: $p_{quad}(5){=}15$, $p_{quad}(6){=}21$, $p_{quad}(8){=}36$, $p_{quad}(12){=}78$. Если «нужно ещё ~столько же» приводит $n$ к величине порядка $2p_{quad}$ при большом $q$ — это симптом несокращённой размерности.

### 5.5.2 Координаты — раздувание I вырожденностью

- [ ] $I = \text{tr}[(X^TX)^{-1}M]$ считается в **псевдокомпонентах / $q-1$ координатах**, не в сырых с $\sum x_i = 1$.
- [ ] Проверка обусловленности: число обусловленности $X^TX$ логируется. Если оно гигантское → $(X^TX)^{-1}$ раздута → I искусственно большой → ложное «нужно много точек».
- [ ] Матрица моментов $M=\int_\mathcal{R} xx^T dx$ берётся по **ограниченной** области (с учётом $L_i,U_i$), а не по полному симплексу.

### 5.5.3 Критерий остановки — относительный, не абсолютный

> Ключевое: $I \propto 1/n$, в ноль не упрётся **никогда**. Абсолютный порог на I → система всегда хочет ещё точек. Это не ответ модели, это отсутствие критерия.

- [ ] Остановка по **относительному выигрышу на точку**, а не по абсолютному значению I:
$$\frac{I_{old} - I_{new}}{I_{old}} < \varepsilon \quad (\varepsilon \approx 2\text{–}5\%)$$
- [ ] Есть **бюджет точек стадии** $N_{max}$ (жёсткий потолок).
- [ ] Есть проверка достаточности: $n \ge p_{quad} + \text{запас}(10\text{–}15) + \text{реплики}$. Augment останавливается, как только она выполнена И выигрыш затух.
- [ ] Остановка срабатывает по **ЛЮБОМУ** из трёх (выигрыш мал / бюджет / $n$ достаточно), не требует всех сразу.
- [ ] В коде НЕТ абсолютного порога вида `while I > threshold: add_point()` — это и есть «протекающий» паттерн.

### 5.5.4 Роль стадии — M5 не доводит до финала

> M5 ставит фундамент (устойчивый старт квадратичной аппроксимации). Прицельную точность добирает M7 под цели веток. Гнать I в пол на M5 = делать работу M7 вслепую, равномерно по области вместо адаптивно.

- [ ] M5 целит в **разумный старт** ($n \gtrsim p + $ запас), а не в финальную точность.
- [ ] Нет дублирования логики: то, что должен делать M7 (адаптивный добор под цель), не зашито в M5 (равномерный добор до низкого I).
- [ ] Понимание зафиксировано в коде/комментариях: «M5 = фундамент, M7 = прицельная достройка». Иначе при ревью соблазн «улучшить» M5 удвоением точек вернётся.

## Блок 5. Суррогаты GP-MoE (M6)

- [ ] Ядро = **Matérn 5/2 ARD** по умолчанию (не RBF). RBF — только опция.
- [ ] **mean = Scheffé-полином**, kernel моделирует остатки (интерпретируемость + честная $\sigma$).
- [ ] **Нижние границы** на $\ell_i$ (≥ мин. межточечное расстояние) и $\sigma_n^2$ (≥ jitter). Проверить, что оптимизатор их соблюдает.
- [ ] **Multiple restarts** при оценке гиперпараметров (marginal likelihood многомодальна). Симптом провала: модель «скачет» между запусками на тех же данных.
- [ ] gating $g_k(x)$ = GMM responsibilities, $\sum_k g_k(x) = 1$ для всех $x$.
- [ ] Прогноз MoE: $\hat{y}=\sum_k g_k \mu_k$. Неопределённость — **два члена**: $\sum_k g_k\sigma_k^2 + \sum_k g_k(\mu_k-\hat{y})^2$. Оба реализованы, не только первый?

## Блок 6. Active learning и ветки (M7)

- [ ] argmax acquisition — **на симплексе с ограничениями**, не свободный градиент в $\mathbb{R}^q$ (candidate set или проекция).
- [ ] Acquisition ветки = EI/desirability **к цели этой ветки**, прогнозы из общей модели.
- [ ] `d_overall(x) = (Π d_i(ŷ_i)^{w_i})^{1/Σw}` — веса и `kind∈{max,min,target}` реально применяются.
- [ ] Новая точка меряется по **ВСЕМ P свойствам**, не только по целевым ветки.
- [ ] Точка после измерения → в общую базу → **переобучение/дообучение общих суррогатов**, доступно всем веткам.
- [ ] **Доля exploration-точек** сохраняется даже внутри ветки (анти-переобучение под узкую цель).
- [ ] **Критерий остановки ветки**: и desirability ≥ порог, И $\sigma$ в точке мала. Не только одно из двух?

## Блок 7. Контроль misspecification (тонкий, часто забывают)

- [ ] Детектор «точка вне всех режимов»: все $g_k(x)$ малы → флаг **возможного нового режима**.
- [ ] Триггер кандидата на **$K{+}1$ эксперта** / перекластеризацию M4 при срабатывании флага.
- [ ] Детектор **стагнации ветки** (EI/прирост desirability ≈ 0 много итераций) → статус «цель недостижима в текущей области».
- [ ] Понимание в коде/комментариях: bias приходит **не от выбора точек**, а от misspecification (выход за освоенные режимы). Это объясняет, почему ветки можно собирать «предвзято».

## Блок 8. Обновление модели и производительность

- [ ] Инкрементальное обновление при фиксированных гиперпараметрах (`O(n²)`) на каждую точку.
- [ ] Полная переоптимизация гиперпараметров (с restarts) — **по триггеру/периодически**, не на каждой точке (иначе $O(n^3)$ каждый раз).
- [ ] При $n$ → тысячи: есть план на sparse GP / inducing points (или явно зафиксировано, что не нужно при текущих масштабах).

## Блок 9. Арбитраж бюджета (если веток > 1)

- [ ] Есть **арбитр**: следующий слот эксперимента → ветке с max нормированным (на приоритет/бюджет) acquisition.
- [ ] Перед исполнением запроса ветки — проверка, не закрыла ли эту зону уже другая ветка (анти-дубль экспериментов).

## Блок 10. Многокритериальная оптимизация (M8)

- [ ] desirability $d_{overall}=(\prod d_i)^{1/m}$ — геометрическое среднее (один $d_i=0$ → весь профиль 0). Проверить, что не арифметическое.
- [ ] Финальный argmax — **мультистартовый** (несвязный прообраз цели).
- [ ] Стоимость учитывается как отдельный критерий desirability.

## Блок 11. Воспроизводимость и состояние

- [ ] Все стохастические шаги (D/I-opt, EM, GP-restarts, EI-старты) принимают **seed**; одинаковый вход → одинаковый выход.
- [ ] `ProjectState` хранит: свойства, **общую базу точек (с тегами происхождения)**, общую модель/гиперпараметры, ветки. Восстановление проекта — полное.
- [ ] `config_snapshot` фиксирует версии/seed/гиперпараметры — снимок воспроизводим.

## Блок 12. Тестовый полигон (верификация)

- [ ] Есть **синтетический оракул с известной истиной** (заданные Scheffé-коэффициенты) — можно измерить, насколько pipeline восстанавливает правду.
- [ ] Golden tests против R: Scheffé (atol 1e-8), GP $\mu,\sigma$ при фикс. гиперпараметрах (atol 1e-6), D/I-эффективность (по `det`, допуск 1-2%, **не точка-в-точку** — алгоритм стохастический!), desirability (atol 1e-8).
- [ ] Edge cases в тестах: $L_i=0$, почти коллинеарные компоненты, вершины симплекса, $n\approx p$, вырожденные ограничения.

## Блок 13. Mixture-Process, schema evolution и augmented design (§13–§14)

> Расширение области/схемы во времени БЕЗ потери уже собранных данных.

- [ ] Точка — **составной версионированный объект** (`DataPoint`: `X` по блокам mixture/process, `Y` мультиотклик, `schema_version`, origin-тег), НЕ плоский массив. Иначе расширение схемы ломает базу.
- [ ] **`MISSING` только в `Y`**, никаких молчаливых 0/средних в `X` или `Y` (новый отклик → старым точкам `Y[new]=MISSING`, суррогат учится лишь на измеренных).
- [ ] Два РАЗНЫХ механизма раскрытия переменных не путаются: **+process (T,P) = append в схему** (`schema_version+1`, миграция старых точек `known-constant(baseline)`); **+mixture (C) = append-компонент симплекса** (Σ переопределяется, старые точки мигрируют на грань `C=0`).
- [ ] **Единый критерий остановки** augment = `stop_reason` из §5.5 (относительный ΔI/I + `n ≥ p+margin` + бюджет), а НЕ второй параллельный критерий. Поиск дубля по коду.
- [ ] Augment: старые fixed-точки **не двигаются**, `X_fixedᵀX_fixed` входит стартовой инфо-матрицей (не ноль); добираются только новые (forward-selection по пулу).
- [ ] Моменты `W` — **аналитическая** закрытая форма по умолчанию (без MC-смещения сэмплера), MC оставлен опцией; mixture-only `W` бит-в-бит == legacy.
- [ ] Регресс «сохранности базы»: mixture-only и augment-без-схемы **бит-в-бит** со старым golden (Scheffé 1e-8, GP μ/σ 1e-6, индексы/`stop_reason`).

## Блок 14. Боевая аугментация и движение границ (§15.0–§15.5)

> Фазовое раскрытие переменных + M8-argmax; разделение «где мерить» vs «где точно».

- [ ] **`x_best` ветки = M8-argmax** по суррогату (`optimize_desirability`, мультистарт + локальное уточнение), а НЕ «лучшая измеренная точка» (иначе рецепт не давит к границе/вершине).
- [ ] §15.0.2: pool (где физически ставим опыты) и `W` (относительно чего оптимально) **легитимно в разных областях**, но обязаны быть в ОДНИХ термах. Assert Условия A (термы) есть; assert «одинаковая область pool/W» — НЕ добавлять (он сломал бы extrapolation-aware I).
- [ ] Высокий I в ограниченных фазах 1–2 — **цена ограничения, не дефект**; падает при раскрытии C (фаза 3: `I_фаза3 < I_фаза2` при той же модели — проверка, что релаксация сработала).
- [ ] Невырожденность `X_poolᵀX_pool` на ограниченном pool: модель не содержит термов, неидентифицируемых на ребре (`cond(...)` / rank-чек активных термов) — это НАСТОЯЩИЙ assert.
- [ ] §15.1.5 атомарное «схема+схема»: один `schema_version+1`, обе причины в `change_log`; `I_atomic ≤ I_seq` (инвариант обоснован, дизайн идентифицируем).
- [ ] Движение границ (`move_bounds`) — корректный примитив для ЛЮБОГО направления (relax/restrict/shift), инвариант Σ=1 для mixture; политика выпадающих fixed-строк (`exclude`/`boundary`/`error`), точки с измеренным `Y` НЕ проецируются.

## Блок 15. Экономический критерий остановки и движение границ (§15.6)

> Двойной стоп (технический И экономический) + денежная триада + A0.7. Аксиомы §0 §15.6 — приоритет над «переоткрытием» из кода.

- [ ] **Калибровка σ GP — фундамент VoI**: предсказанный σ vs реальная ошибка на отложенных точках (premium-угол). Если GP врёт о σ — VoI врёт следом; проверять ДО экономики.
- [ ] **ρ — полноценное GP-свойство** $\rho(A,B,C,T,P)$; `price_изд = price_состав·ρ` (**УМНОЖЕНИЕ**, physics: меньше ρ → легче изделие → дешевле штука; деление в displayed-формуле §3 — описка, знак НЕ менять). Новый ценовой рычаг — process (T,P) через ρ.
- [ ] **Двойной стоп §4**: продолжать, пока `Δd≥ε ∧ d<ceil` И `max_x EI_₽·V·H > c_exp`. `stop_reason ∈ {ceil_reached, stagnation, not_economical}` с приоритетом `ceil_reached > not_economical > stagnation`. Дефолты нейтральны (V=0 ⇒ economic_value=0 ⇒ ведёт себя как чисто технический).
- [ ] **VoI = per-property MC-EI** через РЕАЛЬНУЮ `d_overall` (geo-mean+veto негауссова → MC обязателен, замкнутая формула к агрегату неприменима) + разложение вклада + `justified` (высокий EI принимается, ТОЛЬКО если обоснован разведкой свойства, реально лимитирующего d_overall).
- [ ] **Роли не смешивать**: VoI-движок (d_overall) говорит *КУДА* ставить опыт; экономический слой (цена/триада) — *СТОИТ ЛИ вообще*. `V/c_exp/H` живут атрибутами ветки, НЕ внутри VoI-движка.
