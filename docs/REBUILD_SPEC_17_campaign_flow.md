## §17.0 Мотивация (что было не так)

Главный UI стоял на `PipelineRunner`, который КОНСТРУКТИВНО mixture-only
(`PipelineConfig.region() → SimplexRegion`): процесс-параметров (T, P) нет, M7 —
автоматический active-learning, ветки создаются авто и одноцелевые. Всё, что
требует реальная работа (процесс-параметры с самого старта, эволюция схемы в
любой момент §16.2, РУЧНЫЕ мультицелевые ветки с ролями §16.3, добор точек с
РУЧНЫМ вводом откликов §16.4, экономический стоп §15.6/§16.0.D), уже умеет
`MixtureProcessRunner`, но он жил только в демо-вкладке и МЕРИЛ отклики сам через
синтетический оракул. Итог — два параллельных мира и главный поток на менее
способном движке. §17 сшивает их в один.

Уточнение по фактам (проверено по коду 2026-07-01):
- ввод откликов СУЩЕСТВУЕТ, но на M2 (`_render_m2`, `data_editor` по всем P),
  а не на доборе; точки M5 редактируемой таблицей отклика не сопровождаются;
- процесс-параметров в сайдбар-конфиге нет вообще;
- M7 — `runner.run_m7()` (авто), `render_branches` — mixture-only.

## §17.1 Целевой поток пользователя (контракт UX)

```
1. СТАРТ (сетап проекта):
     • компоненты смеси {A,B,C,…} + границы/baseline (симплекс, Σ=1)
     • ПРОЦЕСС-параметры {T,P,…} + границы (куб, вне симплекса, A0.4) — СРАЗУ
     • отклики (свойства) {strength,gloss,ρ,…}; ρ может быть ценовой ногой
     • строится MixtureProcessRunner с РУЧНЫМ оракулом (§17.2)
2. СТАРТОВЫЙ ДИЗАЙН (seed/M2):
     • предложить точки по составной области (симплекс×куб)
     • показать таблицу точек → пользователь ВРУЧНУЮ вносит отклики по всем P
     • «Заполнить тестовыми» (демо-оракул) — опция для прогонов без лаборатории
3. ВЕТКИ (замена авто-M7):
     • пользователь ВРУЧНУЮ создаёт ветку/ветки
     • в каждой — НЕСКОЛЬКО целей (min/max/target, диапазоны, веса), роли
       (в т.ч. ρ как цель И/ИЛИ цена, §16.0), ограничения области
4. РАБОЧИЙ СТОЛ ВЕТКИ (добор, §16.4 с ручным вводом):
     • предложить N точек (acquisition/argmax по desirability ветки)
     • показать кандидатов → пользователь вносит измеренные отклики
     • долить в ОБЩУЮ базу (origin=branch:{id}, И-1) → переобучить суррогаты
     • пересчитать x*/d_best → §4 двойной стоп (тех+эконом), триада §6 при упоре
5. M8 (рецепт ветки): argmax desirability на составной области = оптимальный
     рецепт {состав + режимы}.
6. ЭВОЛЮЦИЯ В ЛЮБОЙ МОМЕНТ (§16.2): добавить/изменить параметр/компонент/отклик/
     границы → миграция старых точек по ЯВНОЙ политике (A0.6) → пересчёт.
```

## §17.2 Ручной оракул: цикл «предложить → зафиксировать Y» (ядро, Ш1)

**Проблема.** `MixtureProcessRunner.run_branch_round`/`_measure` СЕЙЧАС измеряют
кандидатов сами (через `oracle.evaluate`) — это годится для демо/синтетики, но не
для реальной лаборатории, где отклик приходит от пользователя.

**Решение (logic+test).** Расщепить измерение на два явных шага (A0.6 — мерим
только по команде пользователя):
- `propose_points(branch_id, n, *, explore_frac, n_candidates, seed) -> X_cand`
  — вернуть X-кандидаты (составные координаты) БЕЗ измерения; детерминированно по
  seed; включает exploit-точку (argmax) + explore (acquisition), как
  `run_branch_round`, но без записи в базу.
- `commit_measured(branch_id, X_cand, Y) -> result` — дописать измеренные точки в
  ОБЩУЮ базу с origin-тегом ветки (И-1, без копий), переобучить суррогаты,
  пересчитать x*/d_best. `Y` — матрица n×P в порядке `property_names`; частичные
  строки (MISSING по свойству) допустимы для свойств, где замер невозможен, но
  свойства из целей ветки обязаны быть измерены (иначе — ошибка §17.3).

Демо-оракул сохраняется как удобство: `commit_measured` может получить Y из
`oracle.evaluate` по кнопке «Заполнить тестовыми» — но это ЯВНОЕ действие, не
неявное измерение внутри раунда.

**Гейт-тест (§17.7).** `propose_points` не меняет базу (n_base до/после равен);
`commit_measured` растит базу на n, origin-тег ветки, d_best монотонно не убывает,
суррогаты видят новые точки; повтор с тем же seed воспроизводим.

## §17.3 Валидация «не хватает данных» перед пересчётом (Ш2, logic+test)

Единая проверка (в `CampaignController`) ПЕРЕД любым пересчётом/argmax/стопом —
A0.6/чистота: система не считает молча на дырах, а явно называет, чего не хватает:
- нет измеренных откликов по свойствам, входящим в цели ветки → отказ с перечнем;
- у ветки нет ни одной цели (объектив пуст) → argmax/стоп не определены → отказ;
- ценовая нога объявлена, но `rho_property`/цена состава не заданы → отказ;
- добавлен параметр/отклик без явной политики миграции (§16.2) → отказ.
Возвращает структуру `{ok: bool, missing: [...], text: "..."}`; UI показывает
ошибку и НЕ запускает пересчёт. Гейт-тест: каждая ветка отказа воспроизводима.

## §17.4 Стартовый сетап с процесс-параметрами (Ш3, UI)

Экран старта строит `MixtureProcessRunner` через `begin_phase(mixture_free,
process_free)` + `add_branch` по мере надобности:
- mixture-компоненты (имена, границы, baseline), Σ=1 только на mixture;
- процесс-параметры (имена, [lower,upper] реальные единицы, code↔real в снимок);
- отклики (имена; отметка ценовой ноги ρ и цены состава);
- seed-дизайн (n) → таблица ввода откликов (§17.2 commit_measured вручную).
Реального (не демо) оракула нет — Y приходит от пользователя; демо-оракул
опционален для тестовых прогонов.

## §17.5 UI веток и рабочего стола (Ш4–Ш5)

Переиспользуется уже реализованное (§16.3/§16.4, `campaign_ui.render_campaign`):
редактор целей (`set_desirability`/`set_weights`/`delete_goal`), роли/канал ρ,
рабочий стол. Доработки под реальный поток:
- ручное создание веток (имя, цели, роли, ценовая нога, ограничения);
- рабочий стол переключается с авто-оракула на цикл §17.2 (предложить → таблица →
  ручной ввод Y → долить), «Заполнить тестовыми» остаётся кнопкой;
- показ §4-стопа и денежной триады §6 при упоре в границу (read-only, A0.6).

### §17.6.1 Порядок финала: СНАЧАЛА перенести salvage, ПОТОМ сносить (решение 2026-07-02)

Находка при реализации: salvage-части (ИИ-помощник, проект save/load/delete,
Excel, доли↔части/расход сырья) СЕЙЧАС жёстко завязаны на `PipelineRunner` и
M1–M8-конфиг сайдбара — `assistant.build_context(runner)` читает снимок именно
`PipelineRunner` (стадии M1…M8, benchmark); проект/Excel вызываются только из
M1–M8-рендера. Поток кампании (`render_campaign`) имеет СВОЙ сетап и этих вещей не
использует. Значит «снести M1–M8» и «сохранить salvage» — НЕ одно движение.

Канон («сначала логика+тест, потом UI»; не ломать работающее): финал делается
поэтапно — сперва каждую salvage-функцию ПЕРЕНОСИМ на движок кампании
(`CampaignController`/`MixtureProcessRunner`) с logic+тестом, и лишь потом
удаляем M1–M8 UI и `PipelineRunner` из главного потока.

Шаги миграции salvage (каждый — отдельный коммит, зелёные тесты):
- **C1 (сделано)** — ассистент campaign-native: `assistant.build_campaign_context`
  + `assistant.campaign_ui_guide` строят контекст ПРЯМО из `campaign_overview`
  (без `PipelineRunner`/стадий). Тест `test_iteration20_assistant_campaign.py`.
- **C2** — персистентность кампании: save/load/delete проекта на
  `MixtureProcessRunner` (сериализация схемы+точек+веток+ценовой конфигурации).
- **C3** — выгрузка Excel общей базы/рецепта кампании; доли↔части в сетапе §17.4.
- **C4 (собственно снос)** — `streamlit_app.py`: кампания = ЕДИНЫЙ главный поток;
  удалить `render_stage`/`_render_m2`/`_render_result`/`_render_m3/4`, авто-M7,
  mixture-only `render_branches`, synthetic-benchmark, сайдбар M1–M8 и
  `PipelineRunner` из UI; ассистент/проект/Excel — на перенесённых C1–C3.
